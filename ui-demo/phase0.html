What are Verifiable Credentials?

A verifiable credential is a digital version of a document you already use — a driver's license, a university degree, a land certificate, a vaccination record. The difference is that it's cryptographically signed by the issuer, making it tamper-proof and instantly verifiable by anyone, anywhere.

Think of it as a document that proves itself. No phone calls to the university. No visiting a government office. No waiting days for a background check. The credential carries its own proof.

The three roles: An issuer creates and signs the credential. A holder stores it in their wallet and decides when to share it. A verifier checks its authenticity instantly.

Verifiable credentials are built on open W3C standards, which means they work across different systems, industries, and international borders. This interoperability eliminates redundant verification processes — instead of every new service conducting its own identity check, it can verify a credential already issued by a trusted source.

How Verifiable Credentials Work

The lifecycle follows four steps:

1

Issuance

A trusted entity validates your information and generates a cryptographically signed credential. This is delivered to your wallet.

2

Storage

You store the credential in a digital wallet on your phone, computer, or cloud — entirely under your control, independent of the issuer's systems.

3

Presentation

When you need to prove something, you share the credential — or just the specific claims needed — with a verifier. You choose what to reveal.

4

Verification

The verifier checks the cryptographic signature, confirms the issuer is trusted, and validates the credential hasn't been revoked — all in seconds.

Key insight: The holder controls the data. The issuer doesn't need to be contacted. The verifier doesn't need to store anything. This is fundamentally different from centralized identity databases.

Cost & Benefit

Credentialing infrastructure delivers measurable returns across three dimensions:

For citizens: Eliminate trips to government offices, reduce document fraud, maintain control over personal data, access services faster.

For institutions: Reduce verification costs from days to seconds. VerifyTT reduced issuer onboarding from 1–2 days to minutes. Batch issuance handles 10,000+ credentials per hour.

For governments: Lower administrative costs, reduce fraud, enable cross-sector data sharing, build foundational DPI that scales to new use cases without rebuilding.

The DaaS (DPI as a Packaged Solution) model provides a structured rollout framework — from initial deployment through ecosystem expansion to institutional sustainability — typically across a 24-week engagement.

Standards & Protocols

Active on this screen

W3C Verifiable Credentials Data Model 2.0 — the core standard defining how credentials are structured, signed, and verified.
The credential payload uses JSON-LD with a @context referencing the W3C VCDM 2.0 vocabulary. The credentialSubject block contains claims. Proof is attached via Data Integrity (EdDSA/ECDSA) or secured externally via JOSE/COSE (JWT/SD-JWT). The issuer's DID resolves to a verification method containing the public key.

Issuance Protocol

OID4VCI — OpenID for Verifiable Credential Issuance. Defines how a wallet requests and receives credentials from an issuer.
The wallet sends a token request to the issuer's authorization server, receives an access token, then calls the credential endpoint with a proof of possession. The issuer returns a signed credential. Supports pre-authorized code flow (issuer-initiated) and authorization code flow (wallet-initiated).

Presentation Protocol

OID4VP — OpenID for Verifiable Presentations. Defines how a verifier requests and receives credential presentations from a holder.
The verifier generates a presentation request specifying required credential types and claims. This is encoded as a QR code or deep link. The holder's wallet parses the request, selects matching credentials, optionally applies selective disclosure, and sends a Verifiable Presentation signed by the holder.

Status Check

Bitstring Status List — a privacy-preserving mechanism for checking if a credential has been revoked or suspended.
The issuer maintains a bitstring where each credential maps to a specific bit position. A bit value of 1 indicates revocation. The verifier fetches the status list URL from the credential, checks the relevant bit position. The bitstring is compressed and cached, so verification works offline with stale-but-acceptable data.

Proof Formats Supported

This platform supports five credential standards: W3C-VCDM 2.0, VCDM 1.1, SD-JWT, mDL (ISO 18013-5), and AnonCreds.
Each format has different selective disclosure capabilities. BBS+ and AnonCreds support zero-knowledge proofs. SD-JWT supports claim-level disclosure. mDL supports namespace-level disclosure. VCDM 1.1 and basic Data Integrity (EdDSA/ECDSA) only support full credential presentation.
