# CDPI Agent Service

A standalone agent backend for the CDPI white-label app. Two personas (Senior Technical Architect, Programs & Operations Officer) behind a single webhook, grounded in a RAG corpus over CDPI decks and advisory notes.

Runs on the existing local n8n instance + a new Qdrant container on a shared docker network. Designed to be proxied through the Go app as `/api/agent/chat`, so the browser never talks to n8n directly.

## Architecture (the six-layer skeleton)

```
POST /webhook/agent-chat  →  Webhook (n8n)
                           ↓
                          Config  (loads prompts + extracts message)
                           ↓
                          Router LLM  (Claude Haiku 4.5 — picks persona)
                           ↓
                          Parse Router  (selects persona prompt)
                           ↓
                          Embed Query  (OpenAI text-embedding-3-small)
                           ↓
                          Qdrant Search  (top-5 chunks, graceful on miss)
                           ↓
                          Build Main Request  (persona prompt + retrieved context)
                           ↓
                          Main LLM  (Claude Sonnet 4.6)
                           ↓
                          Respond to Webhook  ({persona, answer, conversationId})
```

Everything is HTTP Request nodes — no langchain node dependencies, portable across n8n versions.

## First-time bootstrap

1. **Fill in `.env`.** N8N key is already there; add the two LLM keys:
   ```
   ANTHROPIC_API_KEY=sk-ant-...
   OPENAI_API_KEY=sk-...
   ```
2. **Bring up Qdrant:**
   ```bash
   docker compose -f docker-compose.agents.yml up -d qdrant
   ```
3. **Attach n8n to the cdpi-agents network** so it can reach `http://qdrant:6333`:
   ```bash
   bash scripts/bootstrap-network.sh
   ```
4. **Ingest the corpus** (reads `../references/` mounted read-only):
   ```bash
   docker compose -f docker-compose.agents.yml --profile ingest run --rm ingest
   ```
   Re-run any time you add or change files in `references/`.
5. **Build and import the workflow** (creates credentials, binds them to nodes, activates):
   ```bash
   python3 scripts/build_and_import.py
   ```
   Re-run whenever you edit a prompt or the template. The script is idempotent — it updates the existing workflow rather than duplicating it.

## Smoke test

```bash
curl -sS -X POST http://localhost:5678/webhook/agent-chat \
  -H 'content-type: application/json' \
  -d '{"message":"What is CDPI'\''s view on collecting birth date at enrolment?"}' | python3 -m json.tool
```

Expected shape:
```json
{
  "persona": "architect",
  "answer": "...",
  "conversationId": null
}
```

## Running the eval

```bash
bash eval/run.sh
```

Writes a timestamped Markdown report to `eval/out/`. First iteration is human-scored on purpose — read the report, note where answers drift from the expected CDPI voice or miss a principle, then tighten the persona prompt and re-import.

## Editing a persona

1. Edit `prompts/architect.md` or `prompts/programs.md` — they are the source of truth.
2. Re-run `python3 scripts/build_and_import.py`. The script rebuilds the Config node's JavaScript with the new prompt content and PUTs the updated workflow.
3. Re-run the eval to see the effect.

## Wiring into the Go app

The eventual hookup: the Go app exposes `/api/agent/chat` and `/api/agent/outputs`, and proxies them to this webhook internally. The browser never sees n8n. Auth/session lives in the Go app.

Minimum Go-side change (conceptual):
```go
// POST /api/agent/chat
// body: { message, conversationId }
// calls http://localhost:5678/webhook/agent-chat
// returns { persona, answer, conversationId }
```

## Known gaps (intentional, to tighten once live)

- **No conversation memory.** Every call is single-turn. When we need multi-turn chat, add a Postgres-backed memory keyed on `conversationId`. The existing walt.id Postgres container is a reasonable host for a new `agents` schema.
- **No automated eval scoring.** Human review first. Once we have a handful of runs we trust, add an LLM-as-judge pass for structural checks (e.g., "does this answer follow the 9–13 word rule for Ask/Why/Scope/Context?").
- **Operations corpus is light.** The Programs persona is aware of this and falls back to general DPI principles with a clear label. Ingest operations notes into `references/` and re-run ingest to close the gap.
- **No `/api/agent/outputs` artifact store yet.** When the Outputs page lands in the Go UI, we add an artifact-persistence node and the table lives in Postgres.
